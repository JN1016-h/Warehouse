# 需求文档

## 简介

本文档定义了仓库管理系统中用户身份管理和财务模块的增强需求。系统需要优化用户角色体系，在界面上持续显示用户身份，并完善财务模块的应收应付款项管理、报表查询和导出功能。

## 术语表

- **系统（System）**: 仓库管理系统
- **用户（User）**: 使用系统的任何人员
- **经销商（Dealer）**: 上下游经销商用户
- **内部员工（Internal_Staff）**: 公司内部工作人员
- **平台管理员（Platform_Admin）**: 具有最高权限的管理员
- **仓库管理员（Warehouse_Admin）**: 负责仓库日常管理的管理员
- **身份标识组件（Identity_Badge）**: 显示当前用户身份的UI组件
- **个人中心（Profile_Center）**: 用户查看和管理个人信息的页面
- **报表（Report）**: 系统生成的数据统计和展示文档
- **应收款项（Accounts_Receivable）**: 系统应该收取的款项
- **应付款项（Accounts_Payable）**: 系统应该支付的款项
- **付款状态（Payment_Status）**: 款项的支付状态（已付款/未付款）
- **Excel导出器（Excel_Exporter）**: 将数据导出为Excel格式的组件
- **PDF导出器（PDF_Exporter）**: 将数据导出为PDF格式的组件

## 需求

### 需求 1: 用户角色定义

**用户故事**: 作为系统架构师，我希望明确定义系统中的用户角色类型，以便系统能够正确识别和管理不同类型的用户。

#### 验收标准

1. 系统应当支持四种用户角色：经销商（Dealer）、内部员工（Internal_Staff）、平台管理员（Platform_Admin）、仓库管理员（Warehouse_Admin）
2. 当用户注册或创建时，系统应当为该用户分配一个且仅一个角色
3. 当查询用户信息时，系统应当返回该用户的角色信息
4. 当用户角色被修改时，系统应当验证新角色是否为有效的角色类型

### 需求 2: 身份标识持续显示

**用户故事**: 作为用户，我希望在使用系统时能够持续看到我的身份角色，以便我清楚地知道我当前的权限和职责。

#### 验收标准

1. 当用户登录系统后，系统应当在页面顶部或导航栏显示身份标识组件
2. 当用户在不同页面间导航时，身份标识组件应当保持可见
3. 当显示身份标识时，系统应当包含用户名称和角色类型
4. 当用户角色为经销商时，身份标识组件应当显示"经销商"标签
5. 当用户角色为内部员工时，身份标识组件应当显示"内部员工"标签
6. 当用户角色为平台管理员时，身份标识组件应当显示"平台管理员"标签
7. 当用户角色为仓库管理员时，身份标识组件应当显示"仓库管理员"标签

### 需求 3: 个人中心身份信息展示

**用户故事**: 作为用户，我希望在个人中心查看我的详细身份信息，以便我了解我的账户状态和权限。

#### 验收标准

1. 当用户访问个人中心页面时，系统应当显示用户的基本信息（用户名、姓名、联系方式）
2. 当用户访问个人中心页面时，系统应当显示用户的角色类型
3. 当用户访问个人中心页面时，系统应当显示用户的账户创建时间
4. 当显示角色信息时，系统应当使用清晰的标签和图标进行视觉区分

### 需求 4: 报表条件查询

**用户故事**: 作为财务人员，我希望能够根据多种条件查询报表数据，以便我能够快速找到需要的财务信息。

#### 验收标准

1. 当用户访问报表页面时，系统应当提供日期范围筛选功能
2. 当用户访问报表页面时，系统应当提供付款状态筛选功能（全部/已付款/未付款）
3. 当用户访问报表页面时，系统应当提供供应商或客户名称筛选功能
4. 当用户输入查询条件并提交时，系统应当返回符合所有条件的记录
5. 当查询结果为空时，系统应当显示友好的提示信息

### 需求 5: Excel报表导出

**用户故事**: 作为财务人员，我希望能够将报表数据导出为Excel文件，以便我能够在本地进行进一步的数据分析和存档。

#### 验收标准

1. 当用户在报表页面点击"导出Excel"按钮时，系统应当生成包含当前查询结果的Excel文件
2. 当生成Excel文件时，系统应当包含所有显示的列（日期、单号、金额、付款状态等）
3. 当生成Excel文件时，系统应当使用清晰的列标题
4. 当Excel文件生成完成时，系统应当触发浏览器下载
5. 当导出的数据超过10000条记录时，系统应当提示用户缩小查询范围

### 需求 6: PDF报表导出

**用户故事**: 作为财务人员，我希望能够将报表数据导出为PDF文件，以便我能够生成正式的财务报告文档。

#### 验收标准

1. 当用户在报表页面点击"导出PDF"按钮时，系统应当生成包含当前查询结果的PDF文件
2. 当生成PDF文件时，系统应当包含报表标题和生成时间
3. 当生成PDF文件时，系统应当使用表格格式展示数据
4. 当生成PDF文件时，系统应当在每页底部显示页码
5. 当PDF文件生成完成时，系统应当触发浏览器下载
6. 当导出的数据超过1000条记录时，系统应当提示用户缩小查询范围

### 需求 7: 应收款项管理

**用户故事**: 作为财务人员，我希望能够查看和管理应收款项，以便我能够跟踪客户的欠款情况。

#### 验收标准

1. 当用户访问应收款项页面时，系统应当显示所有出库订单的应收款项列表
2. 当显示应收款项时，系统应当包含订单号、客户名称、应收金额、付款状态、订单日期
3. 当用户筛选未付款记录时，系统应当仅显示付款状态为"未付款"的记录
4. 当用户更新付款状态为"已付款"时，系统应当记录付款时间
5. 对于任意应收款项记录，系统应当允许授权用户修改付款状态

### 需求 8: 应付款项管理

**用户故事**: 作为财务人员，我希望能够查看和管理应付款项，以便我能够跟踪对供应商的付款情况。

#### 验收标准

1. 当用户访问应付款项页面时，系统应当显示所有入库订单的应付款项列表
2. 当显示应付款项时，系统应当包含订单号、供应商名称、应付金额、付款状态、订单日期
3. 当用户筛选未付款记录时，系统应当仅显示付款状态为"未付款"的记录
4. 当用户更新付款状态为"已付款"时，系统应当记录付款时间
5. 对于任意应付款项记录，系统应当允许授权用户修改付款状态

### 需求 9: 财务金额统计

**用户故事**: 作为财务主管，我希望能够查看财务金额的统计汇总，以便我能够快速了解公司的财务状况。

#### 验收标准

1. 当用户访问应收款项页面时，系统应当在页面顶部显示应收款项总额
2. 当用户访问应收款项页面时，系统应当显示已收款金额和未收款金额
3. 当用户访问应付款项页面时，系统应当在页面顶部显示应付款项总额
4. 当用户访问应付款项页面时，系统应当显示已付款金额和未付款金额
5. 当用户应用筛选条件时，统计金额应当仅计算筛选后的记录
6. 对于任意金额统计，系统应当保留两位小数并使用千分位分隔符

### 需求 10: 付款状态管理

**用户故事**: 作为财务人员，我希望能够更新和跟踪订单的付款状态，以便我能够准确记录款项的收付情况。

#### 验收标准

1. 系统应当支持两种付款状态：已付款（Paid）和未付款（Unpaid）
2. 当创建新的入库或出库订单时，系统应当默认设置付款状态为"未付款"
3. 当用户将付款状态从"未付款"更新为"已付款"时，系统应当记录付款时间戳
4. 当用户将付款状态从"已付款"更新为"未付款"时，系统应当清除付款时间戳
5. 对于任意订单，系统应当在订单详情页面显示当前的付款状态
